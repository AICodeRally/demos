# Demo Standard (Canonical)

Last Updated: 2026-03-28
Scope: all demos under `app/(demos)`

This is the canonical build and quality standard for any new or modified demo.

## 1) Required Demo Contract

Every demo must include:

- `app/(demos)/<slug>/demo.config.ts`
- `app/(demos)/<slug>/layout.tsx`
- `app/(demos)/<slug>/page.tsx` (or clear root redirect behavior)

`demo.config.ts` must define:

- `slug` that exactly matches folder name
- `client.name`
- `product.name`
- `meta.industry`
- `meta.tagline`

Use `defineDemo(...)` from `@/components/shell`.

## 2) Scaffold Workflow

Use the standard starter command:

- `pnpm scaffold:demo <slug> --layout sidebar|topnav|wizard|focus --acts <n>`

Then run:

1. `pnpm generate:registry`
2. `pnpm typecheck`
3. `pnpm verify`

## 3) Layout Policy (Flexible by Design)

Allowed layouts:

- `sidebar` (default)
- `topnav`
- `wizard`
- `focus` (immersive/single-surface demo with minimal shell chrome)

Rules:

- Multiple shell types are first-class; pick layout based on experience goals.
- `topnav` must provide `suite`, `module`, `gradient`.
- `wizard` must provide `wizard.steps`.
- `focus` is preferred for immersive tools that should still inherit theme, toggles, and shared shell standards.

## 4) Navigation and Route Integrity

All nav hrefs in `demo.config.ts` must resolve to an existing `page.tsx` route.

Allowed route categories:

- `nav-visible`: appears in config nav.
- `deep-link`: valid hidden route (dynamic drill-in, modal page, utility page).
- `root`: `/slug` landing route.

Any route that is not nav-visible must be intentionally documented/allowlisted.

## 5) Data and Content Quality

Disallowed in production demo surfaces unless explicitly allowlisted:

- `TODO`, `FIXME`, `TBD`, `WIP`
- Placeholder formulas not marked as intentional mock behavior
- Orphaned mock records with incomplete required fields

Allowed:

- UI input `placeholder="..."`
- Clearly labeled mock/demo datasets with complete internal consistency

## 6) Theming and Visual Consistency

- Use shell theme tokens (`theme`, `colors`) instead of ad-hoc global CSS when possible.
- Limit demo-specific global CSS imports to justified cases.
- Preserve dark/light behavior expectations set by config.

## 7) Registry and Landing Page

- Never manually edit `data/demo-registry.ts`.
- Registry is generated by `pnpm generate:registry`.
- Homepage cards are sourced from generated registry metadata.

## 8) Verification Gates (Required)

For any demo addition/change, run:

1. `pnpm generate:registry --json`
2. `pnpm typecheck`
3. `pnpm verify` (before merge, when feasible)

Recommended additional checks:

1. Nav/route parity script (no missing href targets)
2. Orphan route report with allowlist
3. Placeholder token scan with allowlist

## 9) Definition of Done for Demo Work

A demo change is done only when:

1. Layout choice matches policy (`sidebar/topnav/wizard/focus`).
2. Config metadata is complete and registry-compatible.
3. Nav/routes are coherent (including documented deep-links).
4. No unresolved placeholder debt is introduced.
5. Verification commands pass.
6. Any standard exceptions are documented in the latest audit note.
